## 3. 问题描述

主 spec §5.4 关键事件日历的隐含假设是"以季报为主要数据节奏"——这对半导体、银行、能源等季度数据型公司合适。

但**消费/零售类公司**（Pop Mart、Anta、海底捞、星巴克等）的关键数据节奏是：
- **月度运营更新**（同店销售、新店开业、爆款 IP / SKU 数据）
- **大促期数据**（双 11、618、黑五、圣诞、春节）
- **季报只是月度数据的总结**

撰写 Pop Mart 报告时，§5.4 的"季报"思维明显不够——Labubu 月度销售数据、海外门店开店节奏才是真正的领先指标。

## 4. 建议改法

### 在主 spec §5.4 描述中补充

```markdown
### 5.4 关键事件日历

按公司行业的真实数据节奏组织事件，**不限于季报**：
- 季度 / 半年度 / 年度财报
- 月度运营更新（消费、零售、银行存贷款数据）
- 周度 / 日度高频数据（航运、能源、商品、博彩）
- 大促或季节性窗口（消费品的双 11/618/618/黑五/圣诞；旅游的春节/暑假等）
- 监管 / 政策窗口
- 产品发布或里程碑事件

⚠️ 不要因为模板写"事件日历"就只写季报——按公司真实数据节奏来组织。
```

### 在消费 pack 中补充（如果 fill-consumer-pack proposal 已合并）

```markdown
#### 关键监控数据节奏（消费类）

| 数据 | 频率 | 来源 |
|------|------|------|
| 同店销售月度数据 | 月 | 公司 IR / 月报 |
| 新店开业速度 | 月 / 季 | 公司公告 / 媒体 |
| 大促期销售 | 不定期 | 第三方平台数据 + 公司披露 |
| 库存周转 | 季 | 季报 |
| 海外业务月度更新 | 月 | 公司公告 |
| 季报 / 中报 / 年报 | 季 / 半 / 年 | 交易所披露 |
```
